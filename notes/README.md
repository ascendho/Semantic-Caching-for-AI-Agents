## 二、RAG

简单来说，RAG 就是让大模型在回答问题之前，先去“翻书”（查资料），然后再根据查到的资料来回答。它是目前企业应用大模型最主流的技术方案。以下是详细解读：

### 1. 核心概念：Grounding（锚定/落地）

课件标题提到 **"grounding LLMs with your own data"**。

- **痛点：** 原生大模型（如 GPT-4）只拥有训练时的数据，不知道你的私人文档、公司内部资料或刚才发生的新闻。
- **解决方案：** RAG 把大模型和“你的私有数据”结合起来，让模型的回答有据可依，而不是凭空捏造。

### 2. RAG 的两大核心优势

课件左侧列出了 RAG 解决的两个关键问题：

- **减少幻觉 (Reduce hallucinations)：** 大模型有时会一本正经地胡说八道。通过把相关的“证据”直接塞进模型的上下文（Context Window），强制模型根据证据说话，能显著降低出错率。
- **保持信息新鲜 (Stay fresh)：** 大模型训练一次非常贵且耗时，知识会有截止日期。RAG 允许你在不重新训练模型的情况下，通过实时检索最新的文档（LLM 以前从未见过的数据），让回答保持时效性。

### 3. 工作流程图解

课件右侧的流程图展示了 RAG 的四个步骤：

1. **Question (提问)：** 用户输入一个问题。
2. **Retrieval (检索阶段)：** 系统并不是直接把问题丢给大模型，而是先去 **Document index（文档索引/知识库）** 里搜索。搜索出与问题最相关的 **Relevant documents（相关文档片段）**。
3. **Augmented Generation (增强生成阶段)：**系统把“原始问题”和“搜到的文档片段”组合在一起，形成一个更长的提示词。**LLM（大模型）** 接收到这个组合包，由于它现在手里有了“参考资料”，它就能进行“开卷考试”。
4. **Answer (回答)：** 模型生成最终的、基于事实的回答。

## 二、Agents 对 Token 的巨大消耗

### 1. 核心观点：Agents are token hungry（智能体是 Token 吞噬者）

智能体（Agent）不仅仅是回答一个问题，它是为了完成一个复杂目标而进行的一系列自动化操作。在这个过程中，它会产生海量的 Token 消耗。

### 2. 为什么智能体消耗大？（左侧要点）

- **本性使然 (By nature)：** 一个智能体的工作循环包括：提取信息 (Extract)、制定计划 (Plan)、执行行动 (Act)、自我反思 (Reflect) 以及不断迭代 (Iterate)。
- **多次调用 (MANY LLM calls)：** 为了完成一个任务，智能体可能需要连续给大模型发几十次指令，而不是一次。
- **三个后果：**Consume more tokens： 极其费钱，Token 跑得飞快。**Add additional latency：** 极其耗时，因为每次调用模型、反思、修正都需要时间，用户等待感明显。**Prompts grow in length over time：** 随着对话和步骤增加，为了让模型记住之前的状态，提示词（Context）会变得越来越长，进一步加剧了成本和性能压力。

### 3. 智能体的工作流（右侧流程图）

这部分展示了智能体内部是如何“思考”和“行动”的：

1. **Incoming user request：** 用户输入一个复杂指令（比如“帮我调研并写一份行业报告”）。
2. **Extraction / Planning：** 模型首先拆解任务，制定行动计划。
3. **Generate tasks：** 将大目标拆成一个个具体的小任务。
4. **Loop to solve tasks (循环)：** 智能体进入核心循环。
5. **Agent -> Reflect (反思)：** 每做完一步，智能体会检查一下：“我做得对吗？”“结果符合预期吗？”
6. **Additional steps (迭代)：** 如果发现没做好，或者需要更多信息，它会重新生成新任务再次尝试。
7. **Final response：** 直到任务最终完成，才返回给用户。

## 三、客户支持

### 1. 核心应用场景：加速客户支持

标题指出，这是智能体最常见的用例之一。企业希望利用 AI Agent 来降低 **MTTR (Mean Time To Resolution，平均解决时间)**。也就是说，让客户的问题能更快、更准确地得到解决。

### 2. 左侧示意图：多样化的提问，统一的目标

图中展示了三个不同的用户，他们问法不同，但核心需求是一致的：

- 用户A：“我怎么退款？”
- 用户B：“能把钱退给我吗？”
- 用户C：“你们的退款政策是什么？”
  **Agent 的角色：** 它需要理解这些不同的自然语言背后都是关于“退款”的需求，并给出统一且准确的回复。

### 3. 右侧核心挑战分析（重点）

课件列出了智能体在这一场景下的三个“痛点”：

- **响应速度与用户体验 (UX)：**由于前一张课件提到了 Agent 往往需要多次调用 LLM 进行思考（反思、规划、检索），如果 Agent 运行太**慢**，会让客户在对话框前等很久，从而严重伤害用户体验。
- **FAQ 的堆积：**智能体会生成大量的问答数据。如果这些数据没有得到有效管理，系统会变得臃肿。
- **冗余操作带来的成本危机：**
  - **Redundant agentic RAG operations**：这是最关键的一点。
  - 意思是：如果 1000 个用户问了本质相同的问题（如退款），而智能体**每次**都重新跑一遍完整的“规划 -> 检索文档 (RAG) -> 反思 -> 生成”流程，就会产生大量的**重复计算**。
  - 这会极大推高基础设施的成本（结合第一张课件：推理是瓶颈，非常贵）。

## 四、缓存

### 1. 核心流程：先查缓存，再问模型

课件通过一个闭环流程展示了缓存是如何介入 Agent 与 LLM 之间的：

1. **Check cache for similar queries（检查相似查询）：**当用户向 Agent 提问时，Agent 第一步不是去调 LLM，而是先去问缓存系统（右侧芯片图标）：“以前有人问过类似的问题吗？”**注意：** 这里用的是“similar（相似）”而不是“identical（完全相同）”。在 AI 时代，这通常指“语义缓存”，即意思相近的问题也可以复用答案。
2. **路径 A：Cache hit（缓存命中）：**如果缓存里有答案，直接从缓存返回结果（Respond from the cache）。**结果：** 速度极快（毫秒级），成本几乎为零。
3. **路径 B：Cache miss（缓存缺失）：**如果缓存里找不到，才执行 **Invoke LLM API**（调用大模型 API）。这是那个“又慢又贵”的路径。
4. **Save LLM response in cache（保存结果）：**大模型生成的答案返回后，系统会把它存入缓存，以便下次有人问同样的问题时可以直接使用。

### 2. 为什么这对 Agent 至关重要？

结合前几张课件，缓存解决了以下核心痛点：

- **解决“推理瓶颈”（Slide 1）：** 通过减少对 LLM 的调用次数，缓解了推理算力的压力。
- **解决“Token 饥渴”（Slide 3）：** 大量重复的任务不再消耗 Token，极大地节省了开支。
- **解决“客户支持成本高”（Slide 4）：** 比如 100 个用户问“如何退款”，只有第一个人会产生 LLM 费用，剩下 99 个人都是免费且瞬时响应的。

## 五、传统的缓存技术与语义缓存

### 第一张图：传统（精确匹配）缓存的失败

**标题：Naïve (exact-match) caching fails for natural language**
（天真的精确匹配缓存在自然语言处理中会失败）

- **核心痛点：** 人类表达同一个意思可以用无数种措辞。用户 A 说：*"I want my money back"* (我想要回我的钱)用户 B 说：*"How do I get a refund?"* (我怎么获得退款？)用户 C 说：*"What is your refund policy?"* (你们的退款政策是什么？)
- **传统缓存的局限：**原理： 基于字符串的完全匹配（Exact matches on string data）。**优点：** **Perfect precision**（完美的精准度）。只要匹配上了，答案绝对是对的。**致命伤：** **Poor recall and low cache hit rates**（极差的回放率和极低的命中率）。**结果：** 即使这三个人的意思一模一样，传统缓存也会认为它们是不同的请求，导致 **Cache Miss**（缓存缺失），从而不得不去调用昂贵的大模型。

### 第二张图：语义缓存的引入与权衡

**标题：Enter semantic caching... but with new trade-offs**
（引入语义缓存……但伴随着新的权衡）

为了解决上面的问题，我们需要让缓存“听得懂人话”。

- **语义缓存的原理：**不再比对文字是否一模一样，而是通过**向量化（Vectorized text）**进行语义搜索。它能识别出“退钱”、“退款”、“money back”在数学空间里是靠在一起的。
- **优劣对比表：**

| 特性                   | 传统缓存 (Traditional)         | 语义缓存 (Semantic)                        |
| ---------------------- | ------------------------------ | ------------------------------------------ |
| **匹配方式**           | 字符串完全一致                 | 语义相似度搜索（向量匹配）                 |
| **命中率 (Hit Rate)**  | 极低（文字必须一个字不差）     | **更高 (Higher)**（意思对就行）            |
| **精准度 (Precision)** | 完美 (100%)                    | **有风险**（可能存在误报 False Positives） |
| **结果**               | 经常 **Cache Miss**（费钱/慢） | 经常 **Cache Hit**（省钱/快）              |

- **新的挑战 (False Positives)：**这是语义缓存最大的“副作用”。**例子：** 如果用户问“我怎么修改密码？”和“我怎么重置密码？”，语义缓存可能认为是一样的。但如果业务逻辑中“修改”和“重置”流程不同，缓存就可能给错答案。

## 六、语义缓存是如何运作的

### 1. 向量化用户查询 (Embed user query)

- **动作：** 当用户输入问题（Query）后，第一步不是去查数据库，而是通过 **Embeddings Service（嵌入服务）** 将文本转换成一串数字（即**向量 Vector**）。
- **原理：** 向量就像是文字的“数字坐标”，意思相近的句子在坐标系里的距离会非常接近。

### 2. 相似度比对 (Compare similarity)

- **动作：** 将这个新向量放入 **Semantic Cache（语义缓存库）** 中进行“最近邻搜索”（Nearest-neighbor search）。
- **逻辑：** 寻找缓存中是否已经存在坐标距离非常接近的旧问题。

### 3. 分类判定 (Classify)

- **关键点：** 这里引入了一个 **Threshold（阈值/门槛）**。
- **Cache Hit（缓存命中）：** 如果新问题与旧问题的距离小于阈值（说明极其相似），则直接跳过大模型，**瞬间返回**缓存里的答案。
- **Cache Miss（缓存缺失）：** 如果距离太远，说明这是一个新问题，必须走后面的昂贵流程。

### 4. RAG 流程 (RAG process)

- **动作：** 只有在缓存没命中的情况下，才会启动 **RAG**。
- **过程：** 去 Vector Database（向量数据库）检索知识 -> 喂给 LLM（大模型） -> 生成回答。
- **代价：** 这是最费 Token、最慢的一步。

### 5. 返回回答 (Respond)

- **动作：** 将生成的最终答案发送给用户。

### 6. 更新缓存 (Update cache)

- **动作：** 这是一个**闭环**。系统会将这次“新问题”和“新答案”存入语义缓存库中。
- **意义：** 这样下次再有人问类似问题时，系统就不用再跑第 4 步（RAG）了。它就像一个会不断进化的 FAQ 库。

### 深度总结与技术亮点：

1. **“门卫”角色：** 语义缓存（步骤 1-3）充当了 RAG 系统的“门卫”。它拦截了绝大多数重复或相似的请求。
2. **降本增效：**命中时： 响应时间从几秒缩短到几十毫秒，成本几乎为零。**未命中时：** 虽然多花了一点点向量化（Embedding）的时间，但相对于 LLM 的推理成本可以忽略不计。
3. **智能进化：** 步骤 6 保证了系统随着使用人数的增加，命中率会越来越高，变得越来越“聪明”且“便宜”。

## 七、向量搜索是支柱

### 1. 什么是向量？(Vectors are lists of numbers)

- **外观：** 向量在计算机眼里就是一串长长的数字，例如 [-0.917, 0.370, 0.221, ...]。
- **本质：** 它是数据的**数学表示**。大模型通过“嵌入”（Embedding）技术，将复杂的文字、图片甚至视频，压缩成这些数字。
- **关键点：** 这些数字编码了**“意义与语义” (Meaning & Semantics)**。在数学空间里，意思相近的词（如“猫”和“小猫”），它们的数字坐标会非常接近。意思无关的词（如“猫”和“手机”），它们的坐标会离得很远。

### 2. 图解：坐标系里的秘密

右侧的 3D 坐标系形象地展示了向量搜索的工作原理：

1. **文档（Documents）变成点：** 下方的文档经过 **Embedding** 转换，变成了空间里的一个个彩色圆点。
2. **问题（Question）变成点：** 用户提问后，系统也立即把问题转换成一个向量（黑色的点）。
3. **寻找“最近邻”：**你可以看到黑色的点掉进了一个虚线圈里，旁边有几个橙色和红色的点。**搜索过程：** 计算机不是在“读”文字，而是在算距离。距离黑点最近的那些文档点，就是语义上最相关的答案。**离群点：** 比如右上角的绿点和右下角的蓝点，因为距离太远，被判定为“不相关”。

### 3. 应用场景 (Example applications)

课件列出了向量搜索的四大用途，这些都是大模型产品必备的功能：

- **内容发现 (Content discovery)：** 帮你找到感兴趣的内容。
- **搜索 (Search)：** 即 RAG 的核心，通过语义而不是关键词搜资料。
- **推荐 (Recommendations)：** 比如“买了又买”，也是计算商品间的向量距离。
- **异常检测 (Anomaly detection)：** 如果一个数据点离所有点都很远，它可能就是异常数据。

## 八、现实检查

### 1. 缓存有效性 (Cache Effectiveness) —— “真的有用吗？”

这里关注的是缓存的基本质量：

- **准确率 (Accuracy)：** 语义缓存不是 100% 精确的。系统可能会认为两个完全不同的问题“语义相近”，从而给出一个错误的缓存答案（这就是之前提到的 False Positive）。*挑战：* 我们如何确保返回给用户的缓存结果是正确的？
- **性能 (Performance)：**命中率： 我们的缓存命中率够高吗？如果没人问重复的问题，那缓存就成了摆设。**扩展性与延迟：** 当缓存里有几百万条数据时，进行向量搜索是否还能保持极速？如果查缓存的时间跟调用大模型的时间差不多，那缓存就失去了意义。

### 2. 可更新性 (Updatability) —— “数据过期了怎么办？”

这是缓存管理中最难的部分之一（经典的计算机科学难题）：

- **失效 (Invalidate)：** 如果公司的退款政策变了，原本存在缓存里的“旧政策答案”就变成了错误信息。
- **刷新 (Refresh)：** 我们如何及时删掉旧答案，换上新答案？
- **预热 (Warm)：** 在新功能上线前，能不能提前把一些常见问题的答案塞进缓存，让第一个用户也不用等待？
- *核心矛盾：* 语义缓存不像传统缓存可以按 ID 删除，它是一堆向量坐标，如何精准地清理特定知识点的缓存是一个技术难题。

### 3. 可观测性 (Observability) —— “我们省了多少钱？”

你不能优化你无法衡量（Measure）的东西：

- **指标衡量：** 我们需要实时看到缓存的**命中率 (Hit-rate)**、**响应时间 (Latency)** 以及**节省的成本 (Cost savings)**。
- **质量监控：** 我们需要一套机制来监测缓存是否在频繁给用户提供“驴唇不对马嘴”的答案（Cache quality）。

## 九、如何衡量缓存效果

### 1. 缓存命中率 (Cache Hit Rate, CHR) —— “省钱指标”

- **定义：** 在特定的阈值（Threshold）下，请求直接从缓存中获取答案的频率。
- **核心作用：**它直接决定了**效率**和**成本节省**。命中率越高，意味着你调用昂贵大模型的次数越少，省下的钱就越多，响应速度也就越快。

### 2. 精准率 (Precision) —— “靠谱指标”

- **定义：** 缓存命中的质量。即：在所有缓存给出的答案中，有多少是**真正正确且相关**的？
- **核心作用：**它保证了缓存结果的**可靠性**。如果精准率低，说明缓存经常“张冠李戴”（把不相关的问题当成同一个问题），这会严重伤害用户体验，让用户觉得 AI 在胡说八道。

### 3. 召回率 (Recall) —— “覆盖指标”

- **定义：** 覆盖范围。即：在所有本来可以从缓存中找到答案的请求中，我们实际抓住了多少？
- **核心作用：**它确保了重要的、重复的查询**不会被漏掉**。如果召回率低，说明你的缓存太“保守”了，很多本来可以复用的答案，系统却没识别出来，导致多花了很多冤枉钱去调大模型。

### 4. F1 分数 (F1 Score) —— “平衡指标”

- **定义：** 精准率（Precision）和召回率（Recall）的调和平均数。
- **核心作用：**在语义缓存中，精准率和召回率往往是**鱼与熊掌不可兼得**的（调高阈值会让答案变准，但命中次数会减少；调低阈值会增加命中次数，但答案会变乱）。F1 分数提供了一个**单一的综合指标**，帮助工程师找到那个既省钱又不出错的“最佳平衡点（Sweet Spot）”。

### 深度总结：工程师如何使用这张表？

这张表其实是给工程师的一个**“调优指南”**：

1. **调参数：** 还记得第 8 张课件里的“阈值（Threshold）”吗？工程师会根据这四个指标来调整阈值。
2. **做决策：**如果业务是**金融或医疗**（容错率极低），那么必须优先保证 **Precision（精准率）**，哪怕多花点钱调大模型，也不能给错缓存。如果业务是**闲聊或通用咨询**（容错率高，成本敏感），那么可以适当牺牲一点 Precision，去追求更高的 **CHR（命中率）** 来省钱。

## 十、进阶优化方案

### 1. 提升精准度与召回率 (Precision & Recall)

这部分解决的是：“如何确保缓存给出的答案是真正对的？”

- **距离阈值调优 (Distance Threshold Tuning)：**做法： 调整两个向量被判定为“意思相近”的界限值。**效果：** 找到那个既能挡住重复问题（召回率高），又不会让风马牛不相及的问题撞在一起（减少误报）的平衡点。
- **交叉编码重排序 (Cross Encoder Reranker)：**做法： 向量搜索是“快速但粗略”的筛选。在找到候选缓存后，用一个更精密的“交叉编码器”模型对它们进行二次精准打分。**类比：** 向量搜索是大海捞针选出 10 根针，重排序是找个老师傅拿着放大镜检查这 10 根针哪根最符合要求。
- **LLM 重排序器/验证器 (LLM Reranker / Validator)：**做法： 用一个较小、较便宜的模型来最后把关：“这个缓存答案真的能回答用户的新问题吗？”**效果：** 增加最后一道人工智慧般的“审查机制”，确保准确性。

### 2. 效率与资源利用 (Efficiency & Resource Use)

这部分关注的是如何在保证效果的同时，进一步降低计算开销：

- **模糊匹配 (Fuzzy Matching)：**逻辑： 在进行昂贵的“向量化（Embedding）”计算之前，先进行简单的字符串匹配或拼写纠错。**目的：** 如果用户问的问题跟缓存里的一模一样（或者只是差个错别字），直接拿走，不需要再跑 AI 模型计算向量，从而大幅节省计算资源。

### 3. 环境感知与过滤 (Context & Filters)

这部分解决的是：“如何判断某些问题是否适合使用缓存？”

- **时效性上下文检测 (Temporal Context Detection)：**做法： 识别那些带有时间敏感性的问题（比如“今天天气如何？”“现在的金价是多少？”）。**效果：** 自动避开过时的缓存答案，强制去调大模型获取最新信息，防止“陈年旧案”误导用户。
- **代码检测 (Code Detection)：**做法： 自动识别用户输入的代码片段。**效果：** 代码逻辑极其严密，一个小字符不同意思就全变了，语义缓存很容易出错。对于这类高精细度的内容，直接绕过缓存或采用特殊逻辑处理。

### 总结与串联：

这张课件是 AI 工程师的**“工具箱”**：

- **想省钱：** 就多用“模糊匹配”和“阈值调优”。
- **怕出错：** 就必须加“重排序器（Reranker）”和“验证器”。
- **怕过时：** 就必须开启“时效性检测”。

### 第一张图：实测结果与性能对比

- **沃尔玛的“秘密配方”：**分布式缓存 (Redis)： 保证速度快。**决策引擎 (Decision Engine, DE)：** 这是前一张课件里提到的那些过滤和验证逻辑。**预装 FAQ：** 提前把标准答案塞进去。

### 第二张图：系统架构图

这部分展示了一个大型企业级 AI 缓存系统长什么样：

### 1. 分布式架构与负载均衡（Distributed & Load Balancing）

- **负载均衡器（Load Balancer）：** 当成千上万的用户（Users）同时发起查询时，流量首先进入负载均衡器。
- **多节点（Multiple Nodes）：** 系统由多个“waLLMartCache Node”组成。这样做是为了**横向扩展**：如果流量翻倍，只需要增加更多的节点即可，不会导致系统崩溃。
- **适配器（Adapter）：** 每个节点内都有适配器，负责处理输入数据并将其标准化，以便后续处理。

### 2. 双层缓存存储机制（Dual-tiered Cache Storage）—— 核心创新

沃尔玛将缓存分成了 L1 和 L2 两层，这种设计非常精妙：

- **L1** →→ **向量数据库（Vector Database）：**
  - **作用：** 负责“检索”（Retrieval）。
  - **逻辑：** 这里存的是问题的向量（Embeddings）。当新问题进来时，L1 负责进行**语义搜索**，找出最相似的历史问题。
- **L2** →→ **内存缓存（In-Memory Cache）：**
  - **作用：** 负责“查找”（Lookup）。
  - **逻辑：** 一旦 L1 找到了相似问题的 ID，L2（通常是 Redis 的经典 KV 存储）会以极快的速度直接调取出该 ID 对应的完整答案文本。
- **为什么这么做？** 向量搜索虽然“聪明”但相对重一点，内存查找则极快。通过 L1 找索引、L2 拿数据的配合，达到了性能的最优解。

### 3. 多租户支持（Multi-tenancy）

- **图中表现：** 在右上角的存储桶中，你可以看到数据被切分成了 **Tenant 1, Tenant 2, Tenant 3...**。
- **意义：** 沃尔玛内部有成百上千个应用（比如 App 搜索、后台客服机器人、供应链预测等）。
- **优势：**数据隔离： 不同应用的缓存数据互不干扰，保证安全。**资源共享：** 所有应用共享同一套高性能基础设施，降低了维护成本。

### 总结：这套架构解决了什么问题？

这页课件告诉我们，在真实的企业级环境中，语义缓存不只是一个简单的代码片段，而是一个**复杂的分布式服务**：

1. **高可用性：** 负载均衡和多节点确保了系统不会宕机。
2. **极致性能：** 通过 **L1（向量）+ L2（内存）** 的双层设计，将语义理解的深度与内存读写的速度完美结合。
3. **企业级复用：** 通过**多租户**设计，让这套强大的缓存能力能被全公司不同的 AI 业务同时使用。

## 十二、内存缓存

在沃尔玛（Walmart）的这个架构中，**L2 (In-Memory Cache)** 是整个系统的“极速缓冲层”。简单来说，**In-Memory Cache（内存缓存）** 就是把数据存储在服务器的**运行内存（RAM）**中，而不是硬盘里。常见的技术实现通常是 **Redis** 或 **Memcached**。

### 1. 什么是 L2 内存缓存？

- **物理层面：** 它的读写速度比普通的数据库（甚至比 L1 向量数据库）快得多，通常在**微秒（百万分之一秒）级**。
- **逻辑层面：** 它存储的是**“键值对 (Key-Value)”**。比如：**Key（键）：** 用户问题的哈希值（比如“退款政策”的数字指纹）。**Value（值）：** 之前大模型生成好的答案文本。

### 2. 它有什么用？（核心功能）

在 waLLMartCache 架构中，L2 扮演了三个至关重要的角色：

#### ① 处理“完全一致”的请求（Exact Match Lookup）

这是最省钱的一步。

- 如果用户问了一个**一模一样**的问题（比如 100 个人都问“我的订单在哪？”），系统根本不需要去算复杂的向量（L1 检索），直接在 L2 内存里“秒回”答案。
- **作用：** 极大地节省了 CPU 算力和 Embedding（向量化）的开销。

#### ② 存储 L1 检索的结果（结果快照）

- 当 L1（向量数据库）经过复杂的数学计算，找到了最相似的答案后，系统会顺手把这个结果存入 L2。
- **作用：** 下次如果有人问极其相似的问题，系统可以直接从 L2 取出结果，跳过 L1 的向量比对步骤。

#### ③ 存储“决策引擎”的判定（Decision Snapshot）

- 记得课件里提到的 **Decision Engine (DE)** 吗？它会判断“这个答案准不准”、“是不是过时了”。
- 这些判断逻辑很复杂，但一旦判定通过，系统会将“最终审核合格的答案”存入 L2。
- **作用：** 保证了返回速度。用户不需要等待决策引擎每次都重新“思考”一遍，直接拿现成的“合格品”。

### 3. L1（向量数据库）与 L2（内存缓存）的区别（比喻说明）

为了让你秒懂，我们可以做一个比喻：

- **L1 (Vector Database - 语义层)：** 像是一个**“图书馆”**。它很大，你能通过“意思”在这里找到相关的书（比如你想找关于猫的书，它能把所有关于宠物的书都给你）。**缺点：** 找书（算向量距离）需要时间。
- **L2 (In-Memory Cache - 内存层)：** 像是一个**“桌上的便签条”**。它很小，上面只记录了最常被问到的问题的标准答案。**优点：** 扫一眼就能报出答案，几乎不花时间。

### 4. 为什么沃尔玛要搞“双层”？（Dual-tiered）

如果不搞 L2，只用 L1（向量数据库）：

- **太慢且太贵：** 即使是问一模一样的问题，系统也要经过 Embedding -> 向量计算 -> 排序，这一套下来虽然比调大模型便宜，但对于沃尔玛这种每秒万级流量的公司来说，服务器压力还是太大。

有了 L2 之后：

- **第一道防火墙：** 过滤掉所有重复、简单、刚问过的问题。
- **只有 L2 搞不定的：** 才会去麻烦 L1。
- **L1 也搞不定的：** 才会去麻烦最昂贵的 LLM。

**总结：L2 就是为了把“极速”和“极廉”发挥到极致的那层技术手段。**

## 十三 、决策引擎

### 1. 核心目标：提升缓存精准度

在大模型应用中，为了节省成本和降低延迟，通常会使用“缓存”（Cache）。如果用户的提问（Query）以前有人问过，系统就直接从缓存里拿答案，而不去调用昂贵的 LLM。

传统的缓存匹配主要依靠**语义搜索（Semantic Search）**，即寻找意思相近的问题。但沃尔玛发现这还不够，因为语义相近不代表答案通用。因此，他们引入了“决策引擎”。

### 2. 决策引擎（Decision Engine）的工作流程

图中红虚线框内的部分就是决策引擎，它像一个过滤器，在决定是否查询缓存之前，先进行几项关键判断：

- **第一步：代码检测（Code Detector）**逻辑： 判断用户的提问是否涉及编程代码。**原因：** 代码逻辑极其严谨，差一个字符可能意思全变。语义搜索很难保证代码缓存的完全准确。**流向：** 如果检测到是代码（Yes），**直接绕过缓存**，发给 LLM 处理，以确保代码的准确性。
- **第二步：时效性上下文检测（Temporal Context Detection）**逻辑： 判断提问是否具有时效性（例如：“今天某个商品的价格是多少？”或“现在的库存是多少？”）。**原因：** 缓存里的答案可能是昨天的，而这类问题需要最新的实时数据。**流向：** 如果具有时效性（Yes），**直接绕过缓存**，发给 LLM（或通过 LLM 调用实时插件）处理。
- **第三步：相似度评估器（Similarity Evaluator / 语义搜索）**逻辑： 如果前两步都通过了（即：不是代码，也没有时效性要求），才进入这一步。它将当前的 Query 与“缓存存储（Cache Storages）”中的历史问题进行语义比对。**结果：**缓存命中（Cache Hit）： 发现库里有非常相似的问题，直接从缓存取回 Response（回复）给用户。**缓存未命中（Cache Miss）：** 没找到匹配的，再发送给 LLM 生成答案。

### 3. 总结

这一页课件解释了沃尔玛如何通过**多维度过滤**来优化 AI 系统：

1. **过滤掉不适合缓存的场景：** 排除掉代码类（要求极高精度）和时效类（要求最新数据）的问题。
2. **提高缓存质量：** 只有在通用、稳定的语义场景下才使用缓存。
3. **最终收益：** 既保证了回答的准确性（Precision），又通过缓存降低了 LLM 的调用成本和响应时间。

## 十五、Redis

### 第一页：传统的 Redis —— 高性能数据结构存储

这一页介绍了 Redis 的基本定义和它作为“瑞士军刀”的功能：

- **全称：** **RE**mote **DI**ctionary **S**erver（远程字典服务器）。
- **核心定位：** 一个极其快速的、**基于内存**的键值（Key-Value）存储层。
- **支持的多种数据结构（左侧列表）：** Redis 不仅仅能存简单的字符串，它支持非常丰富的数据类型，每种都有特定的应用场景：**Strings:** 存文本、数字。**Hashes:** 适合存用户信息、配置信息。**Lists:** 常用于消息队列。**Sets & Sorted Sets:** 用于去重、排行榜（带权重的排序）。**Geospatial:** 地理位置索引（如“附近的人”）。**Streams:** 实时事件流处理。
- **总结：** 传统的 Redis 主要被开发者当作**缓存（Cache）和内存数据库**来使用，以极高的性能解决数据读写问题。

### 第二页：现代的 Redis —— AI 时代的向量数据库

这一页展示了 Redis 现在的“新身份”，解释了为什么在 AI 课程里要学它：

- **定义演变：** 它不仅是 KV 存储，现在是一个**快速的二级索引层**。
- **核心新能力：** 支持**向量（Vectors）**、全文搜索、数值、标签等。
- **AI 工作流演示（图示部分）：**非结构化数据： 音频（Audio）、文本（Text）、视频（Videos）。**模型处理：** 通过 AI 模型（如音频模型、文本模型）进行特征提取。**向量嵌入（Vector Embeddings）：** 将这些复杂的数据转化成一串数字（向量）。这些数字代表了数据的“语义含义”。**Redis 的作用（红 R 框）：** Redis 现在可以高效地**存储这些向量**，并支持**相似度搜索**（向量搜索）。
- **总结：** 在大模型应用（如之前提到的沃尔玛案例和智能体）中，Redis 充当了 **“向量数据库”** 的角色。

### 串联总结：为什么要学 Redis？

结合你之前看的沃尔玛案例：

1. **语义缓存：** 沃尔玛用 Redis 存问题和答案。当新问题进来时，先去 Redis 里做“向量相似度匹配”，如果匹配到了，直接给答案。
2. **长期记忆：** 智能体可以用 Redis 存储对话历史和检索到的知识。

## 十六、在 Redis 中实现语义缓存

### 1. 优化的向量搜索（Optimized Vector Search）

这是语义缓存的核心“引擎”。

- **低延迟检索与索引（Low-latency retrieval and indexing）：** 延续了 Redis 内存数据库的特性，在处理数百万条向量数据时，依然能提供毫秒级的响应。这对于实时 AI 应用至关重要。
- **精细化的嵌入向量（Fine-tuned embeddings）：** 支持存储和检索经过微调的向量。这意味着系统能够理解更微妙的语义差异，从而提高缓存的准确性（正如第一页沃尔玛案例所要求的精度）。

### 2. RedisVL SDK

这是开发者的“工具箱”。

- **RedisVL** 是 Redis 官方专门为向量库（Vector Library）开发的 Python SDK。
- **易用的控制（Ergonomic control）：** 它让开发者能够更方便地配置缓存参数，并像操作普通数据库一样进行 **CRUD**（增删改查）操作。它降低了将传统缓存升级为“语义缓存”的开发难度。

### 3. 生命周期与隔离（TTL & Eviction & Namespacing）

这体现了 Redis 作为“成熟数据库”在企业级应用中的优势，是普通向量数据库可能欠缺的：

- **TTL（Time To Live，生存时间）：** 语义缓存不能永久有效（数据会过时）。Redis 支持为缓存设置过期时间，自动清理旧数据。
- **逐出策略（Eviction）：** 当内存满了，Redis 可以根据策略（如 LRU，最久未使用）自动删除不常用的缓存。
- **命名空间与租户隔离（Namespacing & Tenant isolation）：** 这对像沃尔玛这样的大公司非常重要。它可以确保不同部门、不同应用的数据相互隔离，互不干扰，提高了系统的安全性。

## 十七、失败模式

### 1. 质量低（Low Quality）：缓存“没用”或“给错”

这个维度关注的是**效果（Effectiveness）**。

- **命中率问题：** “我们是否以正确的速率命中缓存，从而真正看到成本改善？”*解释：* 如果你的语义匹配门槛设得太严，导致几乎没有命中，那么你为了维护缓存而付出的成本（存储、开发、查询耗时）就白费了，没省下 LLM 的钱。
- **准确性问题：** “缓存返回的回答，对于当前的问题和语境来说是否正确？”*解释：* 这是最危险的情况。如果语义搜索不够精准（门槛太松），系统可能会把一个“看似相似”但其实不同的问题的答案丢给用户，导致 AI 给出错误信息。

### 2. 性能差（Poor Performance）：缓存“太慢”或“带不动”

这个维度关注的是**效率（Efficiency）**。

- **可扩展性：** “对于我们的应用来说，检查缓存是否是一个可扩展的操作？”*解释：* 随着用户量增加，如果缓存查询逻辑设计得不好，它本身可能会崩溃，变成系统的瓶颈。
- **额外开销/延迟：** “缓存检查逻辑是否引入了过多的额外延迟？”*解释：* 使用缓存的目的是为了快。如果你的向量检索加上复杂的决策逻辑（如之前提到的沃尔玛决策引擎）花掉的时间跟直接调 LLM 差不多，那缓存就失去了性能意义。

## 十八、混淆矩阵

### 1. True Positive (TP) —— 理想命中

- **课件描述：** "We hit and we should’ve hit"（我们命中了缓存，而且确实应该命中）。
- **解释：** 这是最完美的情况。用户问了一个之前问过的问题，系统准确地从缓存中找到了完全匹配且正确的答案。
- **价值：** 节省了 LLM 费用，降低了延迟，且保证了回答质量。

### 2. False Positive (FP) —— 错误命中

- **课件描述：** "We hit a wrong cache entry"（我们命中了一个错误的缓存条目）。
- **解释：** 系统认为找到了相似的问题，但实际上缓存里的那个答案并不适用于当前的新问题。
- **风险：** **最危险的情况**。这会导致 AI 给用户提供错误、误导或不相关的答案（即“张冠李戴”）。如果 FP 很高，说明你的语义匹配阈值设得太宽松了。

### 3. False Negative (FN) —— 错过命中

- **课件描述：** "We missed to hit something we could have hit"（缓存里明明有能用的答案，但我们却没命中）。
- **解释：** 系统没能识别出当前问题与缓存中某个问题的语义相似性，结果把请求发给了昂贵的 LLM。
- **影响：** 导致了不必要的资金支出和响应延迟。如果 FN 很高，说明你的语义匹配逻辑太死板或阈值设得太严格了。

### 4. True Negative (TN) —— 正确未命中

- **课件描述：** "We hit and we should not have hit"（*注：此处课件文字疑似有误，按照标准定义应为“我们没命中，且确实不该命中”*）。
- **标准逻辑：** 对于一个全新的、库里完全没有的问题，系统正确地判断出“缓存里没这个”，然后将其发给 LLM。
- **价值：** 保证了系统的准确性，没有强行给用户一个不相关的缓存答案。

### 总结与工程意义

通过这个矩阵，开发者可以对缓存进行**调优**：

1. **追求高 TP（True Positives）：** 最大化性能和成本收益。
2. **严格控制 FP（False Positives）：** 必须确保回答的准确性。如果 FP 增加，就需要引入像**沃尔玛案例**中提到的“代码检测”或“时效性检测”来过滤。
3. **减少 FN（False Negatives）：** 优化向量搜索的算法（如更换 Embedding 模型或调整阈值），让缓存更“聪明”，能识别出更多意思相近的表达。

## 十九、衡量延迟

### 1. 公式解析

公式：WCL=ACL×CHR+(ALL+ACL)×(1−CHR)

这个公式计算的是 **WCL (With Cache Latency)**，即“带缓存情况下的平均系统延迟”。

- ACL **(Average Cache Latency)：平均缓存延迟。**
  - 指检查一次 Redis 缓存需要多长时间。通常非常快（内存操作）。
- CHR**(Cache Hit Ratio)：缓存命中率。**
  - 指有多少比例的请求能直接在缓存里找到答案。
- ALL**(Average LLM Latency)：平均 LLM 延迟。**
  - 指大模型生成答案需要多长时间。通常比较慢（网络传输+推理计算）。
- (1−CHR)**：缓存未命中率。**
  - 即需要去调 LLM 的请求比例。

### 2. 公式逻辑背后的意义

这个公式分为两个部分：

1. **第一部分** ACL×CHR： 命中的情况。用户只需要等待查询缓存的时间。
2. **第二部分** (ALL+ACL)×(1−CHR)**：** 没命中的情况。**注意这里是 `ALL+ACL`**。因为系统是先查了缓存发现没有（花了 `ACL` 时间），然后才去调 LLM（又花了 `ALL` 时间）。

**重要启示：**

1. **缓存延迟 (**ACL**) 越低越好：** 这就是为什么选择 Redis 这种高性能内存数据库的原因。
2. **命中率 (**CHR**) 是关键：** 如果你能通过更好的语义匹配提高命中率，平均延迟会大幅下降。
3. **LLM 的“慢”被中和了：** 即使 LLM 很慢（如 `350ms` 或更久），通过缓存也能在宏观上提升用户体验。

## 二十、衡量成本

### 1. 核心背景：LLM 是按 Token 计费的

大模型（如 GPT-4o）的使用成本非常高，通常分为**输入 Token**（你问的问题）和**输出 Token**（AI 的回答）两部分计费。如果每个请求都发给 LLM，成本会随着用户量增加而飞速上涨。

### 2. 计算器参数拆解（左侧部分）

课件设定了一个典型的企业级应用场景：

- **每日查询量 (Daily Queries):** 10,000 次（一年就是 365 万次）。
- **输入/输出 Token:** 平均每个问题 250 tokens，每个回答 1,000 tokens。
- **缓存命中率 (Cache Hit Rate):** 设为 **35%**。这意味着每 100 个问题里，有 35 个不需要调 LLM，直接从 Redis 拿答案。
- **成本假设 (Cost assumptions):** 按照 GPT-4o 的价格（`2.5/百万输入，2.5/百万输入，`10/百万输出）进行计算。同时算入了 Redis 的存储费和向量生成的费用。

### 3. 成本对比结果（右侧部分）

- **不使用缓存的年成本:** **$38,781.25**这是纯纯交给 OpenAI 等厂商的钱。
- **使用缓存后的年成本:** **$27,320.31**这部分包含了：减去那 35% 的 LLM 支出，再加上一小部分 Redis 存储费和 Embedding（向量化）的费用。
- **最终节省:** **$11,460.94 / 年**结论： 仅仅通过 35% 的缓存命中率，一年就能省下超过一万美元。

### 4. 收益曲线图（底部部分）

底部的图表展示了 **“省钱数额”与“缓存命中率”** 的关系：

- 这是一个线性增长关系。
- **命中率越高，省钱越多。** 例如图中标记点：如果命中率达到 40%，一年就能省下 $13,400。
- 对于日调用量达到百万次的大型企业（如沃尔玛），这个数字可能会变成每年节省数百万甚至上千万美元。
